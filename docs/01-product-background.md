# 产品背景与目标

## 1. 项目背景

当前 Sub2API 的同步工作具有一个很典型的特征：

- 本地仓库不是官方镜像，而是一个持续二开的 fork。
- 二开里既有明确要保留的本地业务语义，也有大量必须回归官方的协议主链。
- 同步时真正难的不是“拉最新代码”，而是“判断某条功能链到底是不是已经和官方对齐”。

在实际工作里，已有工具都能提供部分帮助：

- `git diff` 适合看文件级改动。
- `rg` 适合按字符串找候选点。
- `GitNexus` 适合看调用链、影响面、执行流程。
- `Repomix` 适合把一组文件打包做长文对照。

但这些能力目前是分散的。工程师要靠手工把“功能、链路、差异、风险、决策”自己串起来，代价高，而且容易漏。

## 2. 当前真实痛点

### 2.1 文件级 diff 无法回答功能级问题

举例：

- `Claude /messages/count_tokens` 的行为由路由、handler、service、header 透传、错误映射共同决定。
- `OpenAI /responses/compact` 的一致性不是单看一个函数，而是要同时看：
  - 请求路径 suffix
  - compact body 归一化
  - session / conversation 隔离
  - passthrough 头透传
  - 相关测试是否跟上
- `Gemini native` 的问题往往既涉及 handler failover，又涉及 digest/session 生成，还涉及 `UpstreamModel` 观测字段。

只看单个文件是否不同，并不能自动判断“功能是否同步到位”。

### 2.2 同功能跨语言分散

当前场景不是纯后端，也不是纯前端，而是跨层链路：

- Go：后端网关、service、helper、测试
- Vue/TS：页面、路由、导航入口、API、类型、store、i18n
- SQL：migration
- YAML / JSON / env：配置链
- Markdown：本地二开台账、同步决策文档

如果工具只会对比 Go 文件，它看不到页面入口是否断、DTO 是否失配、配置是否只改了一半。

### 2.3 “允许保留”和“必须官方化”混在一起

当前同步里至少存在两类差异：

- 明确允许保留的本地业务语义
  - 例如本地错误脱敏、中文错误文案、页面保护区、特定业务调度逻辑
- 必须官方化的协议主链
  - 例如 Claude / GPT / Gemini 的请求头、请求体、兼容转换、session、compact、`metadata.user_id`、`UpstreamModel`

真正困难的是让工具知道：

- 什么差异是本地 hook
- 什么差异是同步缺口

这不是普通 diff 工具的职责，但这是同步审计器必须承担的职责。

### 2.4 测试缺口很容易被忽略

官方常常已经补了关键守护测试，但本地同步时：

- 测试文件可能没跟过来
- 测试桩和 helper 可能漂移了
- 工程师肉眼看功能差不多，就先放过去

最后的结果是：

- 代码看起来“差不多”
- 行为层面却和官方已经分叉

## 3. 这次复核暴露出的具体问题

以下不是假设，而是当前同步工作中已经出现的真实类型：

### 3.1 Claude 主链问题

- `metadata.user_id` 生成逻辑仍是本地变体，不是官方当前格式分支。
- session hash 未按官方方式对 `User-Agent` 去噪。
- `count_tokens` 的 URL、header whitelist、header casing 仍与官方不一致。

### 3.2 OpenAI 主链问题

- `/responses/compact` 路径 suffix 没完全跟上官方。
- compact body 归一化规则与官方不同。
- `session_id / conversation_id` 仍未统一按官方做隔离。
- 官方已经有的 session isolation 测试在本地缺失。

### 3.3 Gemini 主链问题

- digest prefix 仍直接混入原始 UA，而不是官方的 UA normalize 版本。
- `UpstreamModel` 观测字段未按官方保留。
- 兼容链与 native 链的错误与响应头逻辑存在偏差。

这些问题说明：同步工作的核心瓶颈已经不是“能不能看到 diff”，而是“能不能按功能链把差异自动归类”。

## 4. 目标用户

ForkTool 主要服务三类人：

### 4.1 同步工程师

负责把官方版本同步到本地二开仓库，希望快速知道：

- 这次更新影响哪些功能链
- 哪些链可以直接回官方
- 哪些链必须人工合并

### 4.2 代码 reviewer / 技术负责人

负责判断同步结果是否可信，希望快速看到：

- 风险最高的链路差异
- 哪些差异命中了本地保护区
- 哪些官方测试本地还没有

### 4.3 后续维护者

在会话中断、人员切换或版本跨度大时，希望快速恢复上下文：

- 这条功能链官方状态是什么
- 本地状态是什么
- 当前同步决策是什么

## 5. 产品目标

ForkTool 的目标不是替代工程师，而是让工程师把注意力放在真正需要拍板的地方。

### 5.1 功能目标

- 以 manifest 为中心管理“功能链”而不是“文件集合”
- 从双仓中提取同一功能的多语言链路节点
- 对关键协议语义做规则化比较
- 输出按严重度排序的同步报告
- 自动标注：
  - `official-required`
  - `keep-local-hook`
  - `manual-merge`
  - `test-missing`
  - `config-chain-check`
- 生成能直接挂到任务系统或 PR 审核里的结果

### 5.2 工程目标

- 单机 CLI 优先
- 以当前技术栈优先落地
- 核心判定尽量确定性
- 允许逐步扩语言，而不是重写核心架构

## 6. 非目标

以下内容本期明确不做：

- 不做图形化 GUI
- 不做“任意语言全自动理解”
- 不做通用代码搜索替代品
- 不做全自动代码合并器
- 不把 AI 作为唯一判定来源

## 7. 成功标准

### 7.1 使用层成功标准

- 工程师能在 5 到 10 分钟内定位一个目标功能链的同步缺口
- reviewer 能直接看到“本轮最危险的 5 条差异”
- 每次同步前后都有结构化报告，不再依赖聊天记忆

### 7.2 质量层成功标准

- 对首批关键链路的误报率可控
- 报告能明确区分“本地 hook”和“官方缺口”
- 缺失的官方测试能被直接指出

### 7.3 组织层成功标准

- 同步流程从“靠熟手经验”转成“工具 + 规则 + 决策”
- 新人接手后不需要先把整个仓库读一遍才能参与同步

## 8. 产品约束

### 8.1 当前仓库优先

ForkTool 的第一版必须优先服务当前 Sub2API 类仓库：

- Go 后端主链
- Vue/TS 前端功能层
- SQL migration
- YAML / JSON / Markdown 配置与文档

### 8.2 必须支持本地决策台账

工具不能只看源码，还必须读取：

- 本地二开差异台账
- 同步决策矩阵
- 功能保护区定义

否则它只能做“找不同”，做不了“给决策”。

### 8.3 必须支持跨仓双输入

核心输入不是一个仓库，而是两个仓库：

- 本地二开仓库
- 官方基线仓库

再加一份 manifest 和一份本地决策文件。

## 9. 核心价值总结

ForkTool 的核心价值不是“看 diff 更方便”，而是：

- 把“同功能全链路是否真正官方化”这件事机器化
- 把“哪些本地语义可以保留”结构化
- 把“同步工作从经验活变成可交接工程流程”
